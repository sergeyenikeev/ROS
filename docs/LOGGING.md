# Логирование PatrolBot

## 1. Общий стиль

В проекте принят единый префиксированный стиль сообщений:

- `[узел]` - запуск, остановка и жизненный цикл;
- `[параметры]` - загрузка и валидация параметров;
- `[база]` - команды движения, watchdog, одометрия;
- `[датчик]` - сообщения mock LiDAR и сенсорного слоя;
- `[nav2]` - отправка целей, feedback, результат action;
- `[миссия]` - переходы state machine патруля;
- `[ошибка]` - критические или отказные сценарии.

## 2. Уровни логов

- `DEBUG` - подробный технический след, нужен при глубокой отладке.
- `INFO` - нормальный рабочий режим.
- `WARN` - подозрительные или пограничные ситуации.
- `ERROR` - ошибка, требующая вмешательства.

## 3. Что логируется

### `patrolbot_base`

- старт и остановка узла;
- загрузка параметров;
- ограничение скоростей;
- срабатывание watchdog по `/cmd_vel`;
- ошибка драйвера моторов;
- публикация odometry/diagnostics в verbose-режиме.

### `patrolbot_mission_manager`

- запуск миссии и источник запуска;
- ожидание action-сервера `Nav2`;
- отправка waypoint;
- feedback action;
- таймаут точки;
- повторы;
- завершение миссии;
- ручная остановка и перезапуск.

### `mock_scan_publisher`

- старт mock-узла;
- публикация `LaserScan` в verbose-режиме.

## 4. Как запускать подробно

Через launch:

```bash
source /home/ubuntu/ROS/scripts/source_workspace.sh
ros2 launch patrolbot_bringup patrol.launch.py \
  map:=/home/ubuntu/ROS/maps/patrolbot_map.yaml \
  mission_file:=/home/ubuntu/ROS/src/patrolbot_mission_manager/config/patrol_route.yaml \
  use_mock_hardware:=false \
  log_level:=debug
```

Через параметр verbose:

- `patrolbot_base/config/base_mock.yaml`
- `patrolbot_base/config/base_hardware.yaml`
- `patrolbot_utils/config/mock_scan.yaml`

можно переключить `verbose_logging: true`.

## 5. Какие сообщения искать при отладке

### Если робот не реагирует на `/cmd_vel`

Искать:

- `[база] команда движения была ограничена`
- `[база] сработал watchdog по /cmd_vel`
- `[ошибка] драйвер моторов не принял команду движения`

### Если patrol не стартует

Искать:

- `[миссия] запрошен запуск миссии`
- `[ошибка] не удалось загрузить маршрут`
- `[миссия] миссия активирована, ожидание action-сервера Nav2`
- `[ошибка] таймаут ожидания action-сервера Nav2`

### Если waypoint не проходит

Искать:

- `[nav2] отправляется цель ...`
- `feedback: distance_remaining=...`
- `[ошибка] таймаут цели`
- `[миссия] ошибка зафиксирована, переход к следующей точке`

## 6. Практические рекомендации

- для первичной диагностики держать `INFO`;
- при анализе нестабильного поведения временно поднимать до `DEBUG`;
- не оставлять `DEBUG` на постоянной основе на Raspberry Pi без необходимости, чтобы не раздувать лог и не тратить CPU на лишний вывод.
